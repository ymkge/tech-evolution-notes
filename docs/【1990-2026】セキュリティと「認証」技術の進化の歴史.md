## ～「合鍵」から「信頼」へ、パスワードのない世界へ～

この資料では、インターネットの普及から現在に至るまで、セキュリティ（特に認証・ログイン技術）がどのように変化してきたかを解説します。

## 第1世代：ID・パスワードと暗号化の夜明け（1990年代）

インターネットが一般に普及し始めた時代です。

### 1990年代前半：Basic認証（IDとパスワードの時代）

- **技術の概要：** IDとパスワードをそのままサーバーに送るだけの非常に単純な仕組み。
    
- **イメージ：** 透明な封筒に合鍵を入れて郵送するようなもの。
    

### 🚨 なぜこれではダメだったのか？

- **【盗聴のリスク】** 通信が暗号化されていないことが多く、途中で誰かに覗き見られると、IDとパスワードが丸見えでした。
    

### 1990年代後半：SSL/TLS（通信の暗号化）

- **技術の概要：** 通信経路を暗号化する技術（今の`https://`）。
    
- **解決したこと：** 配送トラックに鍵をかけて、輸送中に盗み見されるのを防げるようになりました。
    

### 🚨 なぜこれではダメだったのか？

- **【パスワード地獄】** サイトごとにID/パスワードが必要になり、ユーザーが管理しきれず「使い回し」が横行。1箇所から漏れると、他でも不正ログインされるリスクが発生しました。
    

## 第2世代：連携と便利さの追求（2000年代～2010年代前半）

「サイトごとにパスワードを覚えるのは無理」という課題に対し、企業やサービス間での連携が進みました。

### 2000年代半ば：SAML（企業向けシングルサインオン）

- **技術の概要：** 1回のログインで、複数の業務アプリを使えるようにする仕組み。
    
- **イメージ：** 会社の「社員証」を首から下げていれば、社内のどの会議室にも入れる状態。
    

### 2010年頃：OAuth / OpenID Connect（SNSログイン）

- **技術の概要：** 「Googleでログイン」「Facebookでログイン」といった仕組み。パスワードを渡さず、必要な権限（プロフィール情報など）だけをアプリに貸し出す技術。
    
- **イメージ：** ホテルのチェックイン時に、免許証（Google）を見せて、部屋のカードキー（アプリ専用の鍵）をもらう形。
    

### 🚨 なぜこれではダメだったのか？

- **【システム間の鍵管理】** 人間のログインは便利になりましたが、プログラム同士（サーバーとクラウドなど）が連携する際に、「永続的な合鍵（アクセスキー）」をサーバー内に保存する必要がありました。この「保存された鍵」が盗まれる事故が多発しました。
    
- **【パスワードの限界】** パスワード自体がフィッシング詐欺（偽サイト）で盗まれるリスクはなくなりませんでした。
    

## 第3世代：多要素認証と「脱パスワード」（2010年代後半～）

パスワードだけに頼ることに限界が訪れ、「本人確認」の厳格化が進みました。

### 2010年代後半：MFA（多要素認証 / 2段階認証）

- **技術の概要：** パスワードに加え、「スマホのSMS」や「認証アプリ」のコード入力を必須にする仕組み。
    
- **解決したこと：** パスワードが盗まれても、スマホを持っていなければログインできないため、安全性が飛躍的に向上しました。
    

### 🚨 なぜこれではダメだったのか？

- **【面倒くさい】** 毎回スマホを取り出す手間がかかります。
    
- **【MFA疲労攻撃】** 攻撃者が何度も通知を送りつけ、ユーザーが間違って「許可」を押してしまう手法が出てきました。
    

## 第4世代：ゼロトラストと「鍵を持たない」時代（2020年代～現在）

「境界線の内側なら安全」という考え方を捨て、「常に疑い、毎回確認する」時代へ。

### 2020年代前半：FIDO / Passkeys（パスキー）

- **技術の概要：** 指紋認証や顔認証を使い、パスワードを一切使わないログイン方式。
    
- **イメージ：** 鍵穴がなく、生体認証でのみ開く扉。
    
- **解決したこと：** パスワードが存在しないため、フィッシング詐欺で盗まれる情報自体がなくなりました。記憶する必要もありません。
    

### 2021年頃～：Workload Identity連携（キーレスなシステム連携）

- **技術の概要：** （最初の質問にあった技術）プログラム同士の連携において、保存が必要な「合鍵（アクセスキー）」を廃止。代わりに「信頼関係」に基づいて、その場限りの「短期チケット」を発行する仕組みが標準化。
    
- **解決したこと：** サーバーから鍵が盗まれてクラウドが悪用される事故を根本から防げるようになりました。
    

## 第5世代：AIと量子耐性への備え（2024年～2026年・未来）

現在進行形で標準化が進んでいる、最先端のフェーズです。

### 2024年～：コンティニュアス認証（継続的認証）とAI

- **技術の概要：** ログイン時だけでなく、利用中もずっと監視する技術。
    
- **仕組み：** 「マウスの動かし方」「キーボードの叩くリズム」「アクセスする場所」などをAIが分析。「ログインしたAさん、急に操作の癖が変わった？乗っ取られてない？」とリアルタイムで検知し、怪しければその場でロックします。
    

### 2026年頃への展望：耐量子計算機暗号（PQC）への移行

- **課題：** 近い将来、スーパーコンピュータを遥かに超える「量子コンピュータ」が実用化されると、現在使われている暗号（SSL/TLSなど）が解読されてしまう恐れがあります。
    
- **対策：** 量子コンピュータでも解けない新しい数学的な暗号方式への移行準備が始まっています。
    
- **イメージ：** どんなに強力な未来のドリルでも穴を開けられない、新しい素材の金庫へ取り替える作業。
    

## まとめ：進化の全体像

|   |   |   |   |
|---|---|---|---|
|**年代**|**キーワード**|**何が変わったか（例え）**|**当時の弱点**|
|**1990s**|**ID/Pass**|合鍵をそのまま郵送|誰でも見れた・盗めた|
|**2000s**|**Encryption**|鍵を頑丈な箱に入れて輸送|パスワード管理が限界に|
|**2010s**|**OAuth / MFA**|ホテルのカードキー / 2つの鍵|システムの鍵管理が大変|
|**2020s**|**Passwordless / Zero Trust**|生体認証 / その場限りの通行証|(現在普及中)|
|**2026-**|**AI / Quantum**|AIガードマンが常時監視|新しい計算機の脅威|

### 結論

セキュリティの歴史は、**「固定された秘密（パスワードや合鍵）をいかになくすか」**という戦いの歴史です。

Workload Identity連携も、この「固定の鍵をなくす」という大きな歴史の流れの中で生まれた、必然的な技術（システム版のパスキーのようなもの）と言えます。
