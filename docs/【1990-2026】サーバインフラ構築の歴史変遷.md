
#### 1. 【1990年代〜2000年代前半】 物理サーバーの時代

**「土地を買って、自分たちでイチから家を建てる」**

- **どんな時代？**
    
    インターネット黎明期。企業は自社ビルやデータセンターに、物理的な機械（サーバー）を購入して設置していました。
    
- **やり方**
    
    エンジニアがドライバーを持って機械を棚（ラック）に取り付け、ケーブルを手作業で配線し、CD-ROMを入れてOSをインストールしていました。
    
- **なぜ大変だった？**
    
    - **時間がかかる:** サーバーを注文してから届くまで数週間かかります。
        
    - **変更できない:** 一度設置すると、場所の移動や性能の変更が困難でした。
        

#### 2. 【2000年代中盤〜2010年頃】 仮想化とクラウドの登場

**「巨大マンションの一室を借りる（内装は自分で）」**

- **どんな時代？**
    
    「仮想化技術（VMwareなど）」や「初期のクラウド（AWSなど）」が登場。1台の高性能なマシンの上に、仮想的なコンピューターを何台も作れるようになりました。
    
- **やり方**
    
    物理的な配線作業が激減。パソコンの画面上で「作成」ボタンを押すだけで、サーバーが手に入るようになりました。
    
- **なぜ新しい技術が必要だった？**
    
    - **効率化:** 物理サーバーだと「暇な時間」が無駄になりますが、仮想化ならリソースを分け合えます。
        
    - **スピード:** 数週間待ちだったのが、数分でサーバーが用意できるようになりました。
        

#### 3. 【2010年代前半】 構成管理ツールの流行（Chef, Ansibleなど）

**「内装工事の手順書（レシピ）をロボットに渡す」**

- **どんな時代？**
    
    サーバー（箱）はすぐ手に入るようになりましたが、その中身（OSの設定やアプリのインストール）の設定が面倒になりました。サーバーの台数が数十台、数百台と増えたためです。
    
- **やり方**
    
    「このソフトを入れて、設定はこうして」という**手順書（コード）を書くと、ツールが自動でその通りに設定してくれるようになりました。**
    
- **なぜ新しい技術が必要だった？**
    
    - **コピー量産:** 手作業で100台設定すると必ずミスが起きますが、プログラムなら100台でも完璧に同じ設定ができます。
        
    - **「Infrastructure as Code (IaC)」の幕開け:** インフラをプログラムのように扱う考え方が広まりました。
        

#### 4. 【2015年〜2020年頃】 Terraformの台頭（宣言的IaC）

**「完成予想図（設計図）を渡すと、家ごと出現する」**

- **どんな時代？**
    
    クラウド利用が当たり前になり、サーバーだけでなく、ネットワーク、セキュリティ、データベースなど、システム全体が複雑化しました。
    
- **やり方（Terraformの登場）**
    
    これまでは「Aをして、次にBをして」という手順を指示していましたが、Terraformでは**「最終的にこういう状態にしたい（設計図）」**と記述します。すると、現状との差分を計算して、足りない部分だけを自動で作ってくれます。
    
- **なぜ新しい技術が必要だった？**
    
    - **複雑性の管理:** クラウド上の膨大なパーツを管理するには、「手順」ではなく「あるべき姿（設計図）」で管理する方がミスが減るからです。
        
    - **マルチクラウド:** AWSだけでなく、Google CloudやAzureなど、複数のクラウドを統一的に管理する必要が出てきました。
        

#### 5. 【2020年〜2026年】 Pulumi・CDK と AIの融合

**「プログラミング言語とAIで、街全体を動的に生成する」**

- **どんな時代？**
    
    現在から近未来です。インフラはもはや「設定」ではなく、**「本格的なソフトウェア開発」**の一部になりました。
    
- **やり方（PulumiやCDKの台頭）**
    
    Terraformは独自の書き方（HCL）を覚える必要がありましたが、Pulumiなどは普段エンジニアが使っているプログラミング言語（TypeScript, Pythonなど）でインフラを書けます。
    
    さらに、**2023年以降はAI（Copilotなど）がコードを自動生成**する流れが加速しています。
    
- **なぜ新しい技術が必要だった？**
    
    - **開発者体験の向上:** アプリを作るエンジニアが、使い慣れた言語でそのままインフラも作れる方が効率的です。
        
    - **高度なロジック:** 「もし〇〇ならサーバーを2倍にする」といった複雑な条件分岐やループ処理が、従来の専用言語よりも書きやすくなりました。

### 歴史のまとめ比較表

|**時代**|**キーワード**|**家づくりでの例え**|**主な課題（なぜ次へ？）**|
|---|---|---|---|
|**~2005**|**物理サーバー**|木を切って家を建てる|遅い、重い、変更できない|
|**~2010**|**仮想化/クラウド**|マンションを借りる|箱はすぐ来るが、内装（設定）が手作業で大変|
|**~2015**|**構成管理 (Ansible等)**|内装ロボットを使う|内装は楽になったが、建物全体の管理が複雑化|
|**~2020**|**Terraform**|3D設計図から自動建築|独自の書き方を覚えるのが大変、柔軟性が少し低い|
|**Now**|**Pulumi / AI**|プログラムとAIで街生成|アプリ開発と同じ言語・感覚でインフラを扱いたい|

### 結論：なぜ今、PulumiやTerraformなのか？

昔は「サーバーを置くこと」自体が仕事でしたが、今は「ビジネスのスピードに合わせて、柔軟にシステムを変化させ続けること」**が求められています。

- **Terraform** は、「設計図」としてインフラをきっちり管理するのに適しており、現在業界の標準（デファクトスタンダード）です。
    
- **Pulumi** は、さらに一歩進んで、アプリ開発者が「コードを書く感覚」でインフラを自由自在に操るために選ばれています。
    

どちらも、**「人間が手作業でやるミスや時間をゼロにし、創造的なことに時間を使うため」**に進化した道具と言えます。
